Vorteile:

+ Antraege werden innerhalb eines Requests abgearbeitet, d.h. die Antwort
  kommt sofort -> Datenretournierung an EPP-Server ist kein Problem.

  ==> Echtzeit!

  Ausnahme: Stufe mit Bestaetigungspflicht (also Registrartransfer).

+ Da es sich nur mehr um einen Request handelt, fallen die Startup-Phasen
  der verschiedenen Stufen weg (Lade- und Kompilierphase, Datenbankver-
  bindungen aufbauen, TIT-Hash auslesen, etc.).

+ Setzt auf V3 auf, die im Endeffekt nicht extra darauf hinentwickelt
  werden muss.

+ Preforking Servers

+ Preloading

+ Code Caching

+ Persistente Datenbankverbindungen

+ Datenbankverbindung schon beim Serverstart (Childs)

+ Zentrales Logging sowohl von Zugriffen als auch von Fehlern.
  D.h. wenn irgendwo ein Ticket abschmiert, hat man immer die entsprechende
       Perl-Fehlermeldung im Log.

+ Requestuebergreifendes Caching von Daten (pro Child)

+ Apache::Status fuers Monitoring des Ticket-Systems. Vgl. Performance-
  Views in Oracle (z.B. v$instance).

+ Skalierbar mit Bordmitteln: mod_backhand. Bietet Loadbalancing-Mechanismen,
  Failover etc.
